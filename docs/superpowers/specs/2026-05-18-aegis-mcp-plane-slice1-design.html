Goal

The first concrete piece of the vision's MCP plane: a single long-lived FastMCP server, owned by the aegis process, that every spawned Claude Code connects to. Prove the plane is live and callable by exposing one tool, aegis_meta, that briefs the agent on where it is and what aegis offers. Success = an agent in the TUI calls aegis_meta and reports it is on the aegis server.

Locked decisions

  1. Shared HTTP plane now — one FastMCP server, co-resident in the AegisApp asyncio process; all panes' claude subprocesses connect to it (not stdio-per-agent).
  2. Pure FastMCP, native serving — use FastMCP's own run_http_async(host, port) (streamable-http). No http_app() wrapped in FastAPI/uvicorn, no custom ASGI mounting. FastMCP owns the server.
  3. Strict injection — spawned claude gets --mcp-config (aegis only) + --strict-mcp-config; the agent's other MCP servers do not load in aegis sessions (deliberate; the direction toward full tool control).
  4. Auto-prime — --append-system-prompt tells the agent it is inside aegis and to call aegis_meta first.
  5. Additive — built-in tools (Read/Edit/Bash/WebFetch…) are untouched this slice. Wholesale replacement is a later phase (see Feasibility).

MCP server — src/aegis/mcp/server.py

build_server() -> FastMCP returns a FastMCP("aegis") with exactly one tool:

  • aegis_meta() -> str — returns a plain-text orientation briefing: (a) what aegis is (a meta-harness driving Claude Code; you are running inside it); (b) you are connected to the aegis MCP server, which (with strict config) is your only MCP server; (c) the aegis tools currently available — for slice 1, just aegis_meta itself, with a note that more (custom file/web/workflow tools) are coming; (d) guidance: call this once at the start to orient. The briefing body is a module-level constant so it is unit-testable without standing up a server.

Reuses the FastMCP @tool() registration pattern from legacy/server.py (the vision flagged this as the directly reusable scaffolding); fastmcp is already a declared dependency.

Runtime — src/aegis/mcp/runtime.py

AegisMCP owns the server lifecycle, co-resident in the app's asyncio loop:

  • __init__: builds the FastMCP server (build_server()); picks a free port by binding a transient socket to 127.0.0.1:0, reading the assigned port, closing it (accepted benign TOCTOU on localhost). Stores host="127.0.0.1", port.
  • url property → http://127.0.0.1:<port>/mcp/ (FastMCP streamable-http mount path; the exact trailing-path is pinned by the live acceptance test in the plan — the test is the source of truth if FastMCP's default differs).
  • start(): launches server.run_http_async(host=host, port=port) as an asyncio task (stored handle); awaits readiness by polling a TCP connect to (host, port) until it accepts or a short timeout elapses (so a session never spawns before the plane is up).
  • stop(): cancels the task and awaits it (swallowing CancelledError); idempotent.

Injection — drivers/claude.py + cli.py

ClaudeDriver.build_argv(agent, cwd, mcp_url) gains a mcp_url parameter and appends, after the existing flags:

--mcp-config  {"mcpServers":{"aegis":{"type":"http","url":"<mcp_url>"}}}
--strict-mcp-config
--append-system-prompt  <PRIMING>

The --mcp-config value is a compact JSON string (claude accepts JSON strings as well as file paths). PRIMING is a module constant:

“You are running inside aegis, a meta-harness. An MCP server named aegis is attached and is your only MCP server. Call its aegis_meta tool first to learn this environment and the aegis tools available to you, then proceed.”

ClaudeDriver.session(agent, cwd, mcp_url) threads it to build_argv. The make_session closure in cli.py already captures app state; it captures the running AegisMCP.url and forwards it. AegisApp owns the single AegisMCP and passes its url into the session factory it builds.

Data flow & lifecycle

  1. AegisApp.on_mount: construct AegisMCP(), await mcp.start() (server up + ready) before the first pane spawns; store self._mcp; the session factory closes over self._mcp.url.
  2. Each tab spawns claude with the strict aegis mcp-config + priming. The agent (per the appended system prompt) calls aegis_meta; the call hits the shared server; the briefing returns and renders in that pane's transcript as a tool call/result.
  3. All panes share the one server (the plane). Adding tabs adds clients, not servers.
  4. action_quit: close all panes' sessions, then await self._mcp.stop(), then exit.

Error handling

  • Server fails to start (port race, FastMCP error): start() readiness-poll times out → raise a clear error at mount; AegisApp surfaces it as a startup failure and exits non-zero (no half-initialised TUI). The plane is load-bearing under strict config — failing fast is correct.
  • Agent can't reach the server mid-session: that is a normal harness/tool error inside claude — surfaces in the transcript like any tool failure; the pane stays interactive.
  • stop() when never started / double stop: idempotent no-op.
  • Strict-config consequence: the agent's pre-existing MCP servers are absent in aegis sessions. Documented in AGENTS.md and the briefing so it is not a silent surprise.

Verification

  • unit — the briefing constant contains the key orientation phrases (“aegis”, “meta-harness”, “aegis_meta”, “only MCP server”); aegis_meta() returns it.
  • unit — build_argv(agent, cwd, mcp_url) includes --mcp-config with a JSON containing the url under mcpServers.aegis, --strict-mcp-config, and --append-system-prompt with the priming; the existing stream-json flags are unchanged.
  • async runtime — AegisMCP.start() binds a port and a TCP connect to url's host:port succeeds; stop() cancels with no leaked task / no CancelledError escaping; start→stop→start works.
  • acceptance / live (slow, skipped without claude on PATH) — start a real AegisMCP, run claude -p --input-format stream-json --output-format stream-json --verbose --strict-mcp-config --mcp-config <aegis> with the message “call the aegis_meta tool and tell me what server you are on”; assert the streamed events contain a tool_use for the aegis tool and the briefing text in the result. This is the definitive “the plane is on and callable” proof and pins the real mcp-config schema / url path.
  • pilot — with a fake session, an AegisApp mounts with self._mcp started and url reachable; quitting stops it. (Does not require a real claude.)
  • Full hermetic suite (uv run pytest -q -k "not live") stays green; the live driver test is unchanged. Test work written/validated inline.

Tool-replacement feasibility (recorded, not in scope)

The end-goal of replacing the inner harness tools is mechanically supported by claude: --tools "" disables all built-ins, --tools "Bash,Read" selects a subset, --disallowedTools removes specific ones, and --strict-mcp-config (already used here) makes aegis the sole MCP source. So aegis can present its own Read/Write/WebFetch/etc. and disable Claude's. The hard part is reimplementing those tools faithfully; naively disabling Read/Edit/Bash cripples a coding agent. Recommended trajectory: additive now → later, per-agent-permission profiles selectively shadow specific built-ins with aegis equivalents. A dedicated future spec, not this slice.

Non-goals

  • Disabling/replacing any built-in tool (--tools / --disallowedTools untouched).
  • More than the one aegis_meta tool; resources, prompts, workflows, skills plane.
  • Auth / TLS (localhost-only bind; no token).
  • Per-agent or per-tab distinct servers (one shared plane).
  • Persisting the server across aegis restarts; multi-host.
Deferred / next slices

Real aegis tools (vault query, notify, git), an MCP resource for the briefing, the skills/workflow plane, selective built-in shadowing per permission profile, and (much later) the distributed multi-instance plane.
