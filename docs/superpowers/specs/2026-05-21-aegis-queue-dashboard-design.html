Summary

The queue substrate (task-queue v1) is now load-bearing — workers run, callbacks come back, the workflow scaffold composes on top. But the only window into queue activity today is reading JSONL state files by hand. This spec introduces a TUI queue dashboard: an always-on compact strip plus an expandable full-screen dashboard that surfaces queues, in-flight workers, queued tasks, and recent history.

The dashboard is observability-first with a forward seam for a small set of operator verbs (cancel, requeue) later. .aegis.py is treated as a read-only source for queue configuration; editing config from the TUI is out of scope.

Goals

  • Glanceable. Without any keystroke, the user sees a compact one-line strip in every conversation showing queue depth and the most recent in-flight worker.
  • One-keystroke expand. Ctrl+D from any conversation opens a full-screen dashboard. Esc collapses it back. Zero tab cost.
  • Show what is configured. Each queue's agent profile and max_parallel are visible inline with live counts — the user sees the configuration of each queue without leaving the TUI.
  • Live, not polled. Strip and dashboard update on every queue / inbox event, not on a timer. No tick lag.
  • Drill into any task. Arrow-select a row, see its payload, lifecycle timestamps, and result tail (live for running, captured for completed) in an inline right-side detail panel.
  • Jump to live work. When the selected task has a live worker tab in the TUI, one key (>) jumps to it.

Non-goals

  • No config editing. .aegis.py is Python; editing it from inside the TUI is a separate problem. The dashboard reads queues for display only.
  • No operator verbs in v1. Cancel / requeue / pause-queue are deferred. The data model and selection model are designed to admit them without re-architecting (see Future).
  • No inbox visualization. Inboxes are a separate plane; surfacing them is its own surface decision. Out of scope here.
  • No workflow visualization. Workflow runs compose on the queue, so their constituent tasks already appear; the higher-level workflow structure is not surfaced in v1.
  • No web/Telegram surface. TUI only. Telegram already has /sessions; a parallel queue surface there is its own design.

Surfaces

Two surfaces, one data source.

1. The strip — always-on, 1 line

Lives below the conversation content and above the status bar. Visible in every conversation tab. Renders queue depth and the most recent in-flight worker. Adaptive format by queue count:

Queue countFormat
1 queuequeues: tasks ●1/2 ○3 ✓14 ✗2 last: brisk-curie 12s
2–3 queuesqueues: tasks ●1/2 ○3 · impl ●0/1 ○0 last: brisk-curie 12s
4+ queues5 queues · ●3/8 ○12 · ✓42 ✗3 last: brisk-curie 12s
no queues configuredstrip hidden entirely

The strip is one Textual row. On terminals narrower than 80 columns, it auto-truncates the "last:" half before the "queues:" half.

2. The dashboard — Ctrl+D expand, full-screen modal

A Textual ModalScreen overlaid on the current TUI. Four bands stacked vertically on the left ~⅔ of the screen, a detail panel on the right ~⅓.

Bands (top to bottom): QUEUES (config + live counts), IN-FLIGHT (running workers), QUEUED (tasks waiting for a slot), RECENT (last N completed in reverse time order). The cursor lives in IN-FLIGHT by default; ↑↓ moves through every row of all three task bands (queues themselves are not selectable in v1).

Data model

The dashboard's view model is a snapshot built from the live state of QueueManager and the per-worker AgentSession registry (already present in SessionManager).

QueueView

@dataclass
class QueueView:
    name: str            # e.g. "tasks"
    agent: str           # configured agent profile slug
    max_parallel: int
    running: int         # count
    queued: int          # count
    ok: int              # session total
    err: int             # session total

TaskView

One TaskView per visible row in IN-FLIGHT / QUEUED / RECENT. Built from queue.Task records joined with AgentSession when running.

@dataclass
class TaskView:
    task_id: str          # ULID
    queue: str
    state: Literal["queued","running","ok","err","interrupted"]
    payload_summary: str  # first non-empty line of payload, truncated
    worker_handle: str | None   # None if queued
    agent_slug: str | None      # None if queued
    from_sender: str            # producer handle or "telegram"/"substrate"
    enqueued_at: datetime
    dispatched_at: datetime | None
    started_at: datetime | None
    completed_at: datetime | None
    duration: timedelta | None  # live for running, frozen for completed
    result_tail: list[str]      # last N lines of assistant text (running)
                                # OR captured final text (completed)

Strip view

A flat aggregate of all QueueViews plus the most-recently-started running TaskView (the "last:" cell). The strip never holds the full task list.

Event flow

Push-based; no polling. The dashboard subscribes to a single event source, the strip subscribes to a derived aggregate.

Substrate event hook

Currently QueueManager emits a JSONL lifecycle log under .aegis/state/queues/<queue>.jsonl. We add a lightweight in-process observer hook alongside the file write:

# QueueManager
def subscribe(self, callback: Callable[[QueueEvent], None]) -> Unsubscribe: ...

# QueueEvent (one of)
# - enqueued(task_id, queue, payload, from_handle)
# - dispatched(task_id, worker_handle, agent_slug)
# - started(task_id)
# - completed(task_id, outcome, duration, result_text)

Callbacks fire after the JSONL write (so observers see committed state). Hook is fire-and-forget; exceptions in callbacks are logged and swallowed so a buggy observer can't poison the substrate.

Assistant-text tail

For the "result tail (live)" pane, the dashboard subscribes to the running worker's AgentSession stream (the same event stream that ConversationPane consumes). The last N (default 8) lines of assistant text are kept in a ring buffer per TaskView. On completed, the captured final assistant text replaces the live tail.

Strip aggregator

A single QueueDigest service runs in the app, subscribes to QueueManager.subscribe, maintains per-queue counters in memory, and exposes a Textual reactive that both the strip widget and the dashboard's QUEUES band bind to. Snapshots are O(queues) on every event.

What the existing code already gives us

  • QueueManager already tracks queued / running tasks and writes a JSONL lifecycle log.
  • SessionManager already holds a registry of live AgentSessions keyed by handle, including those mounted as background worker tabs.
  • AgentSession already streams AssistantText events — we tap that stream for the live tail.
  • themes.py already exposes a palette role map; both new widgets bind through app.palette like every other widget.

Keybindings

KeyContextAction
Ctrl+Dany conversationopen the dashboard
↑ / ↓dashboardmove cursor through task rows (IN-FLIGHT → QUEUED → RECENT, wrapping)
Enterdashboardfocus / refresh the detail panel for the selected row (no-op if already focused there)
>dashboardjump to the selected task's live worker tab; dimmed/no-op for queued tasks and for completed tasks whose worker tab is gone
Escdashboardcollapse back to the strip

Ctrl+D was chosen over Ctrl+Q (which terminal emulators sometimes intercept as XON/XOFF flow control). If Ctrl+D turns out to conflict with Textual's reserved keys on some terminals, fall back to Ctrl+Y.

Edge cases

  • No queues configured. Strip hidden; Ctrl+D opens a dashboard that says "no queues configured in .aegis.py" and dismisses on any key.
  • Narrow terminal. Strip truncates "last:" half first, then "queues:" half. Dashboard requires a minimum width (~80 cols); below that, render a single-column compact view with no detail panel.
  • Worker tab gone. A completed task's worker tab may have been closed. > is a no-op (and the footer hint dims) when the selected task's worker_handle is no longer in SessionManager.
  • Result tail size. Ring buffer is fixed at 8 lines per task; the detail panel scrolls if more is captured for completed tasks.
  • Restart replay. On boot, QueueManager already marks in-flight tasks as failed:interrupted; QueueDigest picks those up via the same subscribe path during replay. No special-cased replay code in the digest.
  • Tab bar already shows the dashboard "tab". The dashboard is a ModalScreen, not a tab — it does not add a tab cell, but the tab bar header dims while the modal is open (consistent with the AgentPicker modal pattern).
  • Workflow-engine tasks. Workflow runs enqueue through the same queue substrate, so their tasks appear in IN-FLIGHT / QUEUED / RECENT like any other. The workflow itself is not surfaced.

Testing

  • Unit — QueueDigest. Drive a fake QueueManager subscribe stream through enqueued / dispatched / started / completed sequences; assert QueueView[] snapshots after each event. Cover multi-queue + replay (in-flight → failed:interrupted) cases.
  • Unit — QueueStrip renderer. Pure render function from a QueueDigest snapshot to a Rich renderable; assert exact text for the 1-queue / 2–3-queue / 4+-queue / empty cases. No Textual app needed.
  • Unit — QueueDashboard bands. Mount the dashboard against a stub digest + stub SessionManager; assert the visible row sets in IN-FLIGHT / QUEUED / RECENT, the cursor's initial position, and the detail panel for the initially selected row.
  • Behavioural — keybindings. Using Textual's App.run_test, simulate ↑↓ / Enter / > / Esc and assert cursor moves, detail-panel target, modal dismissal, and SessionManager.focus(handle) calls. No live subprocess.
  • Live (marker: live). One round-trip test that enqueues 2 tasks against the real claude binary (or skip when off PATH), opens the dashboard, asserts that the strip aggregates correctly and the detail panel's live tail receives at least one AssistantText event. Matches the existing live test patterns in tests/test_queue_live.py.

Target ≥ 95 % line coverage on queue/digest.py and tui/strip.py; band & dashboard widgets are exercised primarily through behavioural tests.

Future

  • Operator verbs. Add x to cancel a queued/running task (composes with the planned aegis_cancel(task_id) MCP tool from queue v1 polish) and r to requeue a failed task. TaskView already carries the task_id needed.
  • Inbox tier. A fifth band or a sibling modal showing per-handle pending inbox messages; the InboxRouter would need a similar subscribe hook.
  • Workflow tier. Show active aegis workflow runs as collapsed groups whose constituent tasks expand inline. Needs workflow runtime to expose a run registry.
  • Telegram surface. A /queues command in the Telegram frontend backed by the same QueueDigest snapshot, formatted as a chat message. Same data, different transport.
  • Filter / sort. Per-queue filter, per-state filter. YAGNI in v1 with one queue and ≤ ~20 active rows.
