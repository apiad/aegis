Goal

Ship the harness-runs-the-program workflow primitive the vision introduces: a structured Python procedure where the harness drives the sequence of agent interactions, verifies predicates between steps, and retries with feedback on failure. The agent is a participant, not the author — there is no risk of the workflow executing out of order, skipping a step, or finishing early.

V1 ships the engine + one canonical end-to-end workflow (TDD on a plan step) so the substrate has a worked proof, mirroring how the task queue shipped with a live smoke. Workflows are authored as Python @workflow async def functions, auto-discovered via the decorator, and compose on the v1 task queue — workflows do not have their own agent-spawn plane; they ride the queue + inbox primitives.

Locked decisions

  1. Ambition: substrate + one canonical workflow. Ship the engine and a real TDD workflow end-to-end. Proves multi-step, predicate verification, retry-with-feedback, and agent-as-participant in one artifact. A DSL/YAML authoring layer is a deliberate non-goal for v1.
  2. Authoring shape: @workflow async def Python. Async functions decorated with @workflow, taking engine as first positional arg and any other params as keyword args. Workflows live in .aegis.py (inline) or in any module imported by it (typically examples/).
  3. Discovery: decorator-only auto-registry. No explicit workflows = {...} dict. The @workflow decorator registers in a module-level _REGISTRY; name collisions raise ConfigError at load time.
  4. Engine surface mirrors the agent's surface where it makes sense. Workflows speak the same vocabulary as agents (delegate, send, bash (future on the agent side), list_sessions, list_agents) — deterministic Python instead of LLM-driven, same primitives. Where caller constraints differ from the agent side (notably: workflows can block; an agent's turn shouldn't), the workflow surface adds methods agents don't get (drain, spawn, close); the agent-side MCP keeps its existing fire-and-forget aegis_handoff under that name.
  5. Workflows compose on the queue. engine.delegate(queue, payload) is the substrate seam for one-shot worker spawn — internally an aegis_enqueue + await on the callback. No second agent-spawn plane.
  6. engine.caller_handle exposes who invoked the workflow. Set from the MCP from_handle parameter when an agent calls aegis_run_workflow; None when launched from CLI. Workflows that want to run on the calling agent ("implement this doing TDD" said into your own pane → TDD runs on your session) target engine.caller_handle with send. Workflows launched from CLI either engine.spawn(...) a fresh subject or skip the live-agent path entirely.
  7. Live-agent messaging is send (sync, fire-and-forget) + drain (async, await idle). No handoff method on the engine — the agent-to-agent MCP aegis_handoff keeps that name; the workflow's equivalent is just enqueuing into the recipient's inbox. send buffers/wakes through the existing InboxRouter.deliver path (already handles wake-on-idle / mid-turn-buffer / turn-end-flush); drain(handle) waits for the recipient to be idle. Runner auto-drains every touched handle in finally, with a 30s-per-handle ceiling so a hung agent can't trap workflow shutdown. Workflows that need synchronous "tell agent X, wait for them" write engine.send(h, msg); await engine.drain(h) — explicit and composable with multi-message batches.
  8. Two invocation surfaces: CLI + MCP. aegis workflow run <name> [--key=value...] (always sync-block — you typed it, you wait) and aegis_run_workflow(name, kwargs, from_handle, callback=true) MCP tool (mirrors aegis_enqueue exactly — returns workflow_run_id immediately; result lands in producer's inbox tagged sender="workflow:<name>" when done).
  9. No persistence in v1. Workflow runs are in-flight async coroutines; restart loses them. Spawned queue workers and inbox files persist (existing substrate); orphaned in-flight tasks become failed: interrupted via the queue's restart-replay.
  10. Auto-close-on-exit for long-lived agents. The runner closes every handle in engine._spawned_handles in the finally. Workflow can await engine.close(h) explicitly for early retirement.

Architecture

Three modules under src/aegis/workflow/:

ModuleResponsibility
decorator.py @workflow + module-level _REGISTRY: dict[str, WorkflowFn]. Validates signature (async, engine first arg); raises on name collision. Provides list_workflows() + get_workflow(name).
engine.py WorkflowEngine runtime handle every workflow receives. Constructed per-run, bound to live substrate (QueueManager, InboxRouter, AppBridge). Tracks _spawned_handles for auto-close.
runner.py run_workflow(name, kwargs) -> dict — unified entry for CLI + MCP. Builds engine, calls await fn(engine, **kwargs) inside try/except, shapes result as {status, result?, error?}, runs auto-close in finally.

Plus three integration touchpoints:

FileChange
src/aegis/cli.py New aegis workflow run <name> [--key=value...] + aegis workflow list subcommands.
src/aegis/mcp/server.py New aegis_run_workflow MCP tool (BRIEFING gains a paragraph naming it).
src/aegis/mcp/bridge.py AppBridge gains async spawn(profile, *, handle=None) -> str and async close(handle) -> None so the engine can spawn/close long-lived agents through the same bridge the queue uses. Both SessionManager and AegisApp implement them.

Workflow runs share the single live aegis substrate (one MCP server, one QueueManager, one InboxRouter, one SessionManager) — they're just async tasks on the same loop. Concurrent runs work; they share queue caps and the agent pool.

Engine surface

class WorkflowEngine:
    workflow_name: str          # set by runner
    workflow_run_id: str        # ULID, set by runner
    caller_handle: str | None   # MCP from_handle, else None (CLI invocation)

    # ── delegate / live-agent send ────────────────────────────────────
    async def delegate(self, queue: str, payload: str) -> str: ...
    def send(self, handle: str, message: str) -> None: ...
    async def drain(self, handle: str | None = None) -> None: ...

    # ── lifecycle ─────────────────────────────────────────────────────
    async def spawn(self, profile: str, *,
                    handle: str | None = None) -> str: ...
    async def close(self, handle: str) -> None: ...

    # ── predicate / utility / introspection ───────────────────────────
    async def bash(self, cmd: str, *,
                   cwd: str | Path | None = None,
                   timeout: float | None = None,
                   env: dict | None = None) -> CompletedProcess: ...
    def log(self, message: str) -> None: ...
    def list_sessions(self) -> list[SessionInfo]: ...
    def list_agents(self) -> list[str]: ...

    # ── deferred (named for forward-compat; NOT in v1) ────────────────
    # async def ask(self, prompt: str) -> str: ...
    #     Pause the workflow and ask the user a question (Telegram /
    #     TUI prompt); resume with their answer. Human-in-the-loop.

Plus a defined exception:

class WorkflowError(Exception):
    """Expected failure inside a workflow (predicate violated, retry
    exhausted, etc.). Workflows raise this for clean failure reporting.
    Plain Exception is treated as an unexpected crash."""

delegate(queue, payload) -> str — one-shot delegation through the queue substrate

  1. Mint a unique handle f"workflow:{name}:{ulid}" per call.
  2. Bind a tiny _DelegationPromise (session-shaped: async def deliver(msg) resolves an asyncio.Future) under that handle.
  3. queue_manager.enqueue(queue, payload, enqueued_by=handle, callback=True). _handle_of(non-agent-prefix) returns the handle as-is, so the callback routes back to the promise.
  4. await promise — substrate fires inbox_router.deliver on worker finalize.
  5. If msg.status == "error": raise WorkflowError(f"task {task_id} failed: {msg.body}"). Otherwise return msg.body.
  6. finally: unbind handle.

bash(cmd, *, cwd, timeout, env) — async shell

Wraps asyncio.create_subprocess_shell; collects stdout/stderr decoded UTF-8 with replace; returns subprocess.CompletedProcess. Defaults: cwd = find_project_root() or os.getcwd(), timeout = None, env = None. On timeout: kill, await, raise WorkflowError(f"bash timed out after {timeout}s: {cmd}").

spawn(profile, *, handle=None) -> str — long-lived agent

Calls bridge.spawn(profile, handle=handle) (new method). In the TUI the agent appears as a visible ConversationPane tab via the existing _SessionManagerAdapter; in headless it's a SessionManager.spawn. Records the handle in engine._spawned_handles. Returns the handle.

send(handle, message) — sync fire-and-forget

Builds InboxMessage(sender=f"workflow:{name}", body=message, timestamp=now_iso()). Schedules inbox_router.deliver(handle, msg) as an asyncio task (existing deliver already does the right thing: wakes idle sessions, buffers mid-turn, flushes at turn-end). Adds the handle to engine._touched_handles so auto-drain knows whom to wait on at workflow exit. Returns immediately. Unknown handle silently dead-letters into the router's in-memory pending under that handle (documented v1 trait — if you typo a handle, your message lands in a buffer no live session will read).

drain(handle=None) — await idle

Async; awaits each touched handle's session to reach state == ready. If handle=None, drains every handle the workflow has touched via send so far. Per-handle ceiling of 30s; on timeout, emits engine.log warning and returns (don't trap workflow shutdown on a hung agent). Mid-flow drain lets the workflow inject user notes between phases — "write tests" → drain → bash predicate → "fix tests" → drain → bash predicate.

close(handle) — retire a long-lived agent

bridge.close(handle); discard from _spawned_handles. Idempotent.

log(message) — workflow-visible logging

print(f"[workflow:{name}] {message}", file=stderr) + one-line append to .aegis/state/workflows/<run_id>.jsonl (timestamp + message). No clever routing to producer pane / Telegram in v1.

list_sessions / list_agents — read-only introspection

Direct pass-throughs to the bridge methods; same data agents see via MCP tools.

Out of v1 (deliberate)
  • Multi-turn-conversation-with-one-queue-worker. delegate always spawns a fresh worker. Multi-turn drive uses spawn + send/drain + close instead. No fusion primitive.
  • engine.ask(prompt) -> str — human-in-the-loop primitive ("pause the workflow, ask the user something, resume with their answer"). Reserved by the engine surface for forward-compat; concrete implementation (Telegram prompt? TUI modal? both?) is a later spec.
  • Parallel fan-out helper. asyncio.gather(engine.delegate(...), engine.delegate(...)) works natively. No engine.gather.
  • Workflow-spawns-workflow (engine.run_workflow(...)). For v1 a workflow body that needs another workflow imports and awaits the function directly with a shared engine — no engine method needed.
  • Cancellation API beyond cancelling the asyncio task (CLI Ctrl-C or future MCP cancel tool).
  • Polling MCP tool (aegis_workflow_status). Producers either use callback=true (default) or accept fire-and-forget with no recovery.
  • DSL / YAML authoring layer. Python only in v1.

Data flow

CLI invocation

$ aegis workflow run tdd_step --plan-step="VS1 inbox" --test-command="uv run pytest -k inbox"
  1. Typer parses --key=value flags into kwargs: dict[str, str]; loads .aegis.py so @workflow decorators fire and the registry populates.
  2. CLI looks up tdd_step in _REGISTRY. Unknown name → print available workflows + Exit(1).
  3. runner.run_workflow("tdd_step", kwargs) bootstraps a serve-like environment: InboxRouter, QueueManager (from .aegis.py queues), SessionManager, AegisMCP. Binds and starts.
  4. Constructs WorkflowEngine(workflow_name="tdd_step", workflow_run_id=ulid, bridge=bridge, queue_manager=qm, inbox_router=inbox).
  5. try: await fn(engine, **kwargs) — wrapped in the failure-shape logic.
  6. Teardown in finally: close every handle still in engine._spawned_handles; stop the queue manager; stop MCP.
  7. CLI pretty-prints {status, result?, error?} and exits 0 on ok, 1 on error.

MCP invocation

aegis_run_workflow(name="tdd_step",
                   kwargs={"plan_step": "...", "test_command": "..."},
                   from_handle="lucid-knuth",
                   callback=True)
  1. Tool handler validates name exists in registry; missing → {"error": "unknown workflow: 'foo'. Available: [...]"} synchronous return.
  2. Mint workflow_run_id; schedule the workflow as a task via App.run_worker (TUI — establishes Textual active_app context, see queue-v1 lesson) or asyncio.create_task (headless).
  3. Tool returns {workflow_run_id, status: "running"} immediately — producer's tool call doesn't block on the workflow.
  4. On completion: build InboxMessage(sender=f"workflow:{name}", task_id=workflow_run_id, status="ok"|"error", body=<result|error>, timestamp=now). Deliver via inbox_router.deliver(_handle_of(sender_agent(from_handle)), msg). Producer wakes (idle) or buffers (mid-turn), same universal-tagging path as queue callbacks.
  5. callback=False → fire-and-forget; result discarded (no polling in v1).

Failure modes

Unifying principle, same as queue v1: every observable failure is either a startup refusal, a WorkflowError-shaped result, or a tagged inbox message. Never silent.

FailureBehavior
Workflow raises WorkflowError(msg)Result {status: "error", error: msg}. Auto-close fires. Clean exit.
Workflow raises plain ExceptionResult {status: "error", error: f"unexpected: {type(e).__name__}: {e}"}. Traceback to stderr for debug. Auto-close fires.
engine.delegate worker failsdelegate raises WorkflowError(f"task {task_id} failed: {body}"). Workflow catches+retries or propagates.
engine.bash exceeds timeoutKill, raise WorkflowError(f"bash timed out after {timeout}s: {cmd}").
engine.send to unknown handleSilent dead-letter into router's in-memory pending under that handle. Documented v1 trait; a strict-check follow-up can land if it bites.
engine.drain exceeds the 30s-per-handle ceilingEmit engine.log warning, return (don't trap workflow shutdown). Workflow author wraps with asyncio.wait_for if a stricter bound matters.
engine.spawn unknown profileWorkflowError(f"unknown agent profile: {profile!r}. Known: {sorted(...)}") at bridge.
engine.spawn'd agents leaked at workflow endRunner's finally closes everything in _spawned_handles. Logged via engine.log.
Workflow registry name collision at .aegis.py loadConfigError at boot — fail-loud, prints both source locations.
MCP-invoked workflow + producer exited before callbackCallback persists to producer's inbox file (durable). Same dead-letter behavior as queue callbacks.
CLI workflow run with no .aegis.pyFail-loud at config load (existing behavior).
Two aegis processes both running workflowsIndependent MCP servers on different ports, independent state. Per-process scope in v1; coordination is Phase 5 (distribution).

Testing

Hermetic (uv run pytest -q -m "not live")

  • @workflow decorator — registers under function name; raises TypeError if not async def; raises TypeError if first param isn't engine; raises ConfigError on name collision.
  • WorkflowEngine.delegate — stub QueueManager + stub worker returning "OK"; assert delegate returns "OK". Error path: stub worker fails → WorkflowError with task_id in message. Concurrent calls get unique inbox handles.
  • WorkflowEngine.bash — real shell for trivial commands (echo, pytest --version); timeout test with sleep 5 and timeout=0.1.
  • WorkflowEngine.spawn / close — stub SessionManager + FakeHarness; spawn adds the handle to _spawned_handles; close removes it + idempotent.
  • WorkflowEngine.send / drain — stub InboxRouter records every deliver call; assert send queued a properly-tagged InboxMessage (sender workflow:<name>, body, timestamp); drain returns when every touched session is state == ready; drain timeout path triggers log warning + returns (no raise).
  • engine.caller_handle — engine constructed with caller_handle="h" exposes it; engine constructed with caller_handle=None (CLI path) exposes None.
  • Auto-close on workflow exit — workflow spawns 2 agents, returns; assert both got bridge.close() in the runner's finally.
  • runner.run_workflow — WorkflowError → {status: error, error: msg}; plain Exception → {status: error, error: "unexpected: ..."}; success returning "x" → {status: ok, result: "x"}.
  • CLI — aegis workflow list enumerates registry; aegis workflow run <unknown> exits non-zero with available list; aegis workflow run <known> calls the runner end-to-end (stub bridge so no real subprocess).
  • MCP — aegis_run_workflow with callback=True returns workflow_run_id immediately; after completion, InboxMessage(sender="workflow:<name>", task_id=<run_id>, status="ok") lands in producer's inbox. Unknown-name returns documented error shape synchronously.

Live (tests/test_workflow_live.py, marker live)

One workflow that does a tiny real TDD loop: write a test asserting 1 + 1 == 2, verify it fails on a stub module, write the implementation, verify it passes. Uses real claude subprocesses via the queue. Skips when claude not on PATH. Bounds: ~90s budget, sonnet + full permission matching the queue smoke. Proves the full chain: CLI → runner → engine → delegate → real worker → bash predicate → success path.

Non-goals (v1)

  • DSL / declarative authoring (YAML, JSON). Python @workflow only.
  • Multi-turn-with-one-queue-worker primitive. Use spawn + send/drain + close on a long-lived agent instead.
  • Workflow persistence / resume across restart. Runs are in-memory asyncio tasks.
  • Workflow-spawns-workflow as an engine method. Direct await of an imported function is the v1 answer.
  • Cancellation MCP tool. asyncio.CancelledError (CLI Ctrl-C) suffices for v1.
  • Polling MCP tool (aegis_workflow_status). callback=true (default) is the canonical recovery path.
  • Dynamic workflow registration at runtime (e.g. via MCP). Discovery is .aegis.py-load-time only.
Open questions deferred
  • Predicate primitives library. v1 workflows do their own predicates in Python (if "PASS" in proc.stdout: ...). A shared library of reusable predicates (assert_tests_fail(cmd), assert_file_exists) may be worthwhile once 3+ workflows show common patterns.
  • Workflow tab in the TUI. v1 shows queue workers spawned by a workflow as their natural tabs; the workflow itself has no dedicated surface. A "workflow tab" showing live progress + log lines may help once multiple long workflows run concurrently.
  • MCP aegis_bash tool for agents. The engine's bash shape is the reference implementation for what agents will eventually get. Out of scope here; will land when agents need it.
