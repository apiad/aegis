Goal

Make it possible to drive an autonomous coding session from Telegram: spawn agents, message them, switch between them, and get notified when each finishes — from a phone, against a long-lived headless aegis daemon. Not full TUI parity; a focused remote control surface.

Locked decisions

  1. Headless daemon is primary. New aegis serve runs with no Textual, long-lived (VPS). The TUI becomes one of several frontends, not the home of the sessions.
  2. DRY core from day one (1a). Extract a frontend-agnostic AgentSession + SessionManager; rewire ConversationPane to delegate to it now. No duplicated turn/metrics/state loop ever exists.
  3. AppBridge is the seam. SessionManager is the concrete AppBridge (the Protocol at mcp/bridge.py:16-21 already matches it); the MCP runtime binds to it instead of to AegisApp.
  4. Unified root resolution. find_project_root walks up from cwd to the nearest ancestor with a .aegis.py; that dir is the harness cwd for every spawned agent. One discovery rule for aegis, aegis init, aegis serve — replaces today's default_search_paths().
  5. Single-chat, sticky active agent. /<handle> switches the sticky target; bare messages go to it; /<handle> <text> one-shots without moving the pointer.
  6. Notifications: final reply text only. Telegram sink keeps AssistantText blocks; drops AssistantThinking/ToolUse/ToolResult. No tool digest.
  7. Per-turn status message, edited in place. One message per turn carrying the same status-line fields as the TUI StatusBar; refreshed ~2s; flips to done/error; then the final reply is posted, chunked.
  8. Raw async Bot API client. Thin httpx wrapper over ~4 endpoints; no python-telegram-bot framework dependency.
  9. Auth: single hardcoded chat id. Updates from any other chat are dropped. No registration flow.
  10. Agents run with their configured permission. Aegis does not override permission; whatever the .aegis.py profile sets is used as-is.

Data flow

Inbound (Telegram → agent)

  1. getUpdates long-poll returns an update; resume offset stored.
  2. Auth: chat.id != telegram_chat_id → log at debug, drop.
  3. Parse: leading / → command (or /<handle> routing); else bare text → active agent.
  4. Target agent working and command is a send → lucid-knuth is working — /interrupt first. /interrupt, /close always allowed; pure-query commands never touch a turn.
  5. AgentSession.send(text) → lazy-start harness on first turn → turn task begins.

Outbound (agent → Telegram), per turn

  1. on_state(working): sendMessage → ⏳ lucid-knuth · working · ·opus·high ↑0 (–) ↓0; store the returned message_id (as an awaitable) keyed by (handle, turn); start a refresher task.
  2. Refresher edits that message every ~2s with the live status line from SessionMetrics (same fields as the TUI StatusBar). Telegram same-message edit limit ≈1/s; 2s is safe.
  3. Sink accumulates AssistantText.text blocks only.
  4. on_state(finished): cancel refresher; final editMessageText → ✅ lucid-knuth · ready · …final metrics… (or ⚠️ … · error).
  5. Post accumulated reply as new message(s), MarkdownV2-escaped, split ≤4096 chars, labeled lucid-knuth (1/3); hard cap (e.g. 5 chunks) then … (truncated, N more chunks).

Error handling

  • Harness error / exit-without-Result. Existing pane.py:166-175 logic moves into AgentSession: commit metrics, state → error. Telegram: status final-edits to ⚠️ … · error; any partial AssistantText collected is still posted.
  • Bot API failures. Every call retry-with-backoff. getUpdates network error → exponential backoff (cap ~30s), resume from last offset (no missed/duplicate updates). sendMessage/editMessageText failure → log + drop (never crash a turn); next 2s refresher self-heals an edit miss. 429 → honor retry_after.
  • Status-message race. Turn finishes before the status sendMessage returns → finish-edit awaits the stored message_id; if it never arrives, fall back to a fresh sendMessage for the final state. Per-turn message_id + refresher; /interrupt and /close cancel the refresher cleanly.
  • Stale sticky pointer. Active agent closed (/close or harness death) → fall back to most-recently-active remaining, else none.
  • Non-auto permission stall. A profile whose permission isn't auto can hit a permission prompt unanswerable from Telegram → turn stalls on working. Known v1 limitation; /interrupt is the escape hatch. Documented, not induced.
  • Auth. Chat-id check before any routing; mismatch logged at debug, dropped.
  • Secret hygiene. Token never logged; env AEGIS_TELEGRAM_TOKEN takes precedence over .aegis.py and is the documented VPS path.

Testing

TDD per repo convention: failing test first, commit per logical unit; fast hermetic suite is uv run pytest -k "not live". All test-writing stays inline (not delegated) — it is the verification layer, especially for the TUI rewire.

  • AgentSession (hermetic). Fake HarnessSession yielding scripted Events (existing parser fixtures). Assert: lazy start on first send; observers fire in order; metrics commit on Result; interrupt/close cancel + set state; harness-error and exit-without-Result → error. Also pins TUI behavior post-rewire.
  • ConversationPane regression. Existing TUI tests stay green after the 1a rewire — the safety net. Add any missing coverage before rewiring.
  • SessionManager. spawn/list/close/ interrupt; isinstance(mgr, AppBridge); handoff (existing bridge tests repointed from AegisApp).
  • Telegram routing (bulk, hermetic). Fake Bot API client (records outbound, feeds scripted getUpdates). Table-driven over every command incl. /new <bad-slug>, /close, working-guard, wrong-chat-id drop, no-active-agent, sticky-switch vs one-shot. Assert exact outbound sequence: status send → refresh edits → final edit → labeled chunked reply with truncation cap.
  • format.py. Pure-function tests: status-line render, MarkdownV2 escaping, chunking + cap.
  • find_project_root. tmp_path trees: cwd, ancestor, none (init path), nested closest-wins.
  • aegis serve wiring. One hermetic test: no-Telegram-config comes up headless, MCP binds to SessionManager, SIGTERM → clean close_all. No live claude in CI; existing tests/test_integration_live.py left as-is.

Non-goals

  • Permission-prompt handling over Telegram (agents run their configured permission; non-auto stalls are a documented limitation).
  • Forum/topic-per-agent addressing (single linear chat in v1).
  • Multi-chat / multi-user / registration flow (single hardcoded chat id).
  • Webhooks (long-poll only — no inbound port/TLS on the VPS).
  • HTTP control-plane / separate bridge process (Approach 2 — the documented scale-out path, not v1).
  • Tool-activity digests in notifications (final reply text only).
  • Per-spawn working-directory selection (one project root per daemon, resolved from .aegis.py).
Open questions & deferred items
  • Hard chunk cap value (placeholder: 5 chunks then truncation marker) — tune after first real use; uncapped mode is a later toggle if wanted.
  • Refresher interval (placeholder ~2s) — may raise if Telegram rate-limits in practice.
  • TUI rebind already done in this spec (1a); the remaining deferred item is whether AegisApp itself should later be a thin frontend over SessionManager (today it still owns the pane list) — out of scope here.
