Aegis Task Queue (v1) — Design

Goal

Give Aegis a first-class inter-agent delegation primitive: agent A enqueues a task on a named queue; the substrate spawns a worker of the queue's configured agent profile, runs the task, and (if the producer asked for a callback) delivers the result into the producer's inbox as a normal incoming message. Cron/recurring schedules and migration of the vault/+/jobs/ substrate are explicit non-goals for v1 — the primitive earns its keep by enabling something Aegis cannot do today, and those phases build on top later.

V1 also introduces the agent inbox primitive that makes callbacks possible. Inboxes are general-purpose: callbacks, Telegram pushes, agent-to-agent handoffs, and future long-lived-terminal notifications all converge on the same delivery channel with universal sender tagging.

Locked decisions

  1. Use case. V1 is the inter-agent delegation primitive, not a replacement for the vault job substrate. Vault-job migration is a later phase (vision Phase 6).
  2. Two primitives, not one. Task queues and agent inboxes are separate concerns that compose. The callback path goes through the inbox primitive; it is not a separate return channel.
  3. Substrate-deterministic dispatch. Queues are not LLM-controlled. Each queue's worker policy is static config: one agent profile + a max-parallel cap. No priority, no routing logic, no agent-chosen worker.
  4. Async + callback (no sync block). aegis_enqueue returns a task_id immediately. If callback=true, the result is delivered into the producer's inbox when ready. Producer keeps working in between.
  5. Wake-on-idle inbox semantics. An inbox arrival wakes an idle agent into a new turn. Mid-turn arrivals buffer and flush at the next turn boundary. Multiple pending messages flush as a single batched user-message block (each retains its own header). Aegis sessions become event-driven.
  6. Universal tagging. Every inbox message and every queue entry carries a typed sender tag (queue:<id>, telegram, agent:<handle>, system; forward placeholders like terminal:<name> are reserved in the schema) and a timestamp. Callbacks additionally carry task_id + status.
  7. File-persisted state. Append-only JSONL under .aegis/state/queues/<queue>.jsonl (events: enqueued / dispatched / completed / failed) and .aegis/state/inboxes/<handle>.jsonl (one record per delivered message). Both gitignored. Schema versioned with a v: 1 field per record.
  8. At-most-once workers. On aegis serve restart, every in-flight task is marked failed: interrupted and a failure callback is delivered. Workers are not resumed. Producer is responsible for idempotency / retry.
  9. Ephemeral workers. One task per spawned agent; the worker exits when the task completes. No worker pools, no reuse.
  10. Day-one consumers. Agents-via-MCP (any spawned Claude Code session), the TUI (worker tabs surface automatically via the existing SessionManager → ConversationPane path), and Telegram (inbox arrivals already render as chat lines via the existing frontend). No new TUI queue-status pane in v1 — visibility is "queues are visible via the worker tabs they spawn, full stop"; JSONL files are trivially tailable for debugging.
  11. Cancel via inbox, not a dedicated tool. A producer wanting to stop a runaway worker uses aegis_handoff to post a cancel-request into the worker's inbox. Keeps the MCP surface minimal.

Data model

Three records, all JSONL-serialisable. Every record carries v: 1.

# Static config (from .aegis.py)
Queue:
  name: str               # "impl", "review", "research", ...
  agent_profile: str      # references an agent profile declared in the same .aegis.py
  max_parallel: int       # cap on concurrent workers; tasks beyond cap wait FIFO

# A task — append-log event types: enqueued | dispatched | completed | failed
Task:
  id: str                       # ULID; sortable + URL-safe
  queue: str
  payload: str                  # prompt the worker receives as its first user message
  enqueued_by: SenderTag        # "agent:lucid-knuth" | "telegram" | "system" | ...
  enqueued_at: iso8601
  callback: bool                # if true, deliver result into enqueued_by's inbox
  status: pending | dispatched | completed | failed
  worker_handle: str | None     # set on dispatch
  result: str | None            # set on completion (worker's final assistant text)
  error: str | None             # set on failure (short reason)
  completed_at: iso8601 | None

# An inbox message — what lands in an agent's inbox
InboxMessage:
  sender: SenderTag             # typed; see Locked decisions §6
  timestamp: iso8601
  body: str                     # the message text the agent sees
  task_id: str | None           # set when sender starts with "queue:"
  status: ok | error | None     # set when sender starts with "queue:"

SenderTag is a small typed union encoded as a string prefix (queue:<id>, agent:<handle>, telegram, system, plus reserved future prefixes). The substrate validates known prefixes; unknown but well-formed prefixes are accepted as opaque to keep schema migrations cheap.

The TUI requires no new code: ConversationPane already renders any AgentSession, and a queue-spawned worker is just another session. The existing per-tab state-dot / sticky * / bell mechanism fires automatically when an inbox message lands on a producer's session. Tab handles get a queue suffix (e.g. vivid-laplace · impl#01J42) so workers are distinguishable at a glance.

Telegram likewise needs no new commands: an inbox arrival on a Telegram-bound handle renders as a chat line via the existing TelegramFrontend, with the substrate-rendered header (> from queue:impl · task#01J42 · ok · 2026-05-20 07:14) baked into the message body.

MCP surface

Two new tools on the slice-2 MCP plane. Both follow the existing from_handle convention (the agent's priming bakes its own handle, which it passes back as a parameter).

ToolParametersReturns
aegis_enqueue queue: str, payload: str, callback: bool = true, from_handle: str {task_id: str, queued_position: int} — queued_position is 1-indexed position in the queue's pending FIFO at enqueue time (1 = head; ignores in-flight workers).
aegis_task_status task_id: str {status, result?, error?, completed_at?, queued_position?}

What is intentionally not exposed.

  • No aegis_inbox_peek — auto-flush at turn-boundary is the canonical delivery; mid-turn introspection breaks the reasoning invariant.
  • No aegis_cancel — producers cancel via aegis_handoff into the worker's inbox.
  • No aegis_create_queue — queues are static config in .aegis.py; the MCP plane does not mutate the queue topology.

Internal refactor. aegis_handoff becomes a thin wrapper around InboxRouter.deliver(to_handle, InboxMessage(sender=agent:<from_handle>, body=payload, timestamp=now)). Universal tagging means handoff payloads now arrive with the same sender · timestamp header that callback results use. No semantic change visible to producers; the handoff tool's parameters and return shape are unchanged.

Config: .aegis.py

Append a top-level queues dict to the existing config module.

# .aegis.py
agents = {
  "claude-impl":   {"harness": "claude", "model": "opus",   "permission": "auto"},
  "claude-review": {"harness": "claude", "model": "sonnet", "permission": "ask"},
}

queues = {
  "impl":   {"agent": "claude-impl",   "max_parallel": 2},
  "review": {"agent": "claude-review", "max_parallel": 4},
}

Validation at aegis serve boot: every queue's agent must reference a declared profile. Unknown agent → fail-loud, print which queue is broken. Same pattern as the existing profile loader.

Data flow

One delegation, end to end:

  1. Enqueue. Producer lucid-knuth calls aegis_enqueue(queue="impl", payload="implement plan X", callback=true). The MCP tool forwards to QueueManager.enqueue("impl", "implement plan X", enqueued_by="agent:lucid-knuth", callback=true).
  2. Persist + ack. QueueManager assigns ULID 01J42, appends an enqueued record to .aegis/state/queues/impl.jsonl, pushes onto the in-memory FIFO. Returns {task_id: "01J42", queued_position: 1} synchronously. Producer's tool call returns; the producer continues working.
  3. Dispatch. The per-queue dispatch loop sees 0 in-flight, cap 2, FIFO non-empty. Pops the task. Calls SessionManager.spawn(profile="claude-impl", opening_prompt="implement plan X"). Gets back worker handle vivid-laplace. Appends a dispatched record (sets worker_handle). Registers a completion observer on the worker's AgentSession.
  4. TUI surfaces the worker. Tab vivid-laplace · impl#01J42 appears in the tab bar automatically — no TUI code change needed. Alex can click in to watch the worker run live, or ignore it.
  5. Worker runs. Streams turns through its pane normally. Exits when its work is complete (worker decides — either via its own aegis_meta close call or by reaching a natural stop). Substrate captures the worker's final assistant text as result, appends a completed record.
  6. Callback. Substrate builds InboxMessage(sender="queue:impl", task_id="01J42", status="ok", body=<result>, timestamp=now). Calls InboxRouter.deliver("lucid-knuth", msg). Router writes-through to .aegis/state/inboxes/lucid-knuth.jsonl and pokes the producer's AgentSession.
  7. Receive. Producer is idle → arrival wakes it into a new turn whose user-message is the drained inbox. (Alternatively: producer is mid-turn → message buffers, flushed at next turn boundary.) The user-message begins with a substrate-rendered header: > from queue:impl · task#01J42 · ok · 2026-05-20 07:14, followed by the worker's result text.
  8. Continue. Producer reads the result. Decides next move (enqueue another task, hand off, respond to Alex, exit). Loop continues.

Variants.

  • Multiple producers, one queue: strict FIFO. No priority in v1.
  • Multiple arrivals while a producer is mid-turn: batched into one user-message block at turn end, each entry retains its own header. Senders may be heterogeneous (e.g. one from queue:impl, one from telegram) — rendered as a list.
  • Producer is idle, two arrivals land within milliseconds of each other: both buffered; the wake schedules one turn with both batched, not two turns.

Error handling

Unifying principle: every observable failure becomes either a startup refusal or a tagged inbox message. The substrate never silently drops a producer's expected callback.

FailureV1 behavior
Worker exits abnormally (subprocess crash, harness error)Substrate appends a failed record; delivers InboxMessage(status="error", body=<reason>) to the producer's inbox.
Worker hangs / runs too longNo automatic timeout in v1. Operator interrupts via the TUI's existing Esc binding or Telegram /interrupt <handle>. A task_timeout config knob is straightforward and deferred to v1.1.
aegis serve restart mid-flightOn boot: replay queue logs; mark every dispatched-but-not-completed task as failed: interrupted; deliver a failure callback into the producer's inbox file (durable on disk even if the producer's live session is also gone). Workers are not resumed.
Producer has exited by the time the callback firesOrphaned inbox file at .aegis/state/inboxes/<handle>.jsonl. Callback persists for forensics; no live delivery. Documented v1 trait — generated handles are per-spawn unique, so a closed session's inbox is effectively dead-lettered.
Queue at capNew tasks FIFO-wait. No load shedding, no priority bumping. queued_position in the enqueue response tells the producer its depth.
Unknown queue name on enqueueMCP tool returns an error synchronously; the agent sees it as the tool result. No queue side-effects.
Invalid .aegis.py (queue references unknown agent profile)aegis serve refuses to start, prints which queue is broken. Same fail-loud pattern as the existing profile loader.
.aegis/state/ not writableRefuse to start. TUI, Telegram, and MCP plane all stay down until the operator fixes it.

Testing

Continue the existing hermetic-vs-live split (per AGENTS.md: uv run pytest -q -k "not live" for the fast suite).

Hermetic unit tests

  • QueueManager — FIFO ordering; cap enforcement; dispatch loop wakes on headroom; persistence-then-replay across a simulated restart correctly marks in-flight tasks as failed: interrupted. Uses a stub SessionManager that records spawn() calls without launching subprocesses; lets us simulate worker completion + failure deterministically.
  • InboxRouter — deliver() writes through to JSONL + buffer; live-session delivery triggers AgentSession.deliver; idle-session delivery schedules a wake-turn; drain() returns batched messages in arrival order; persistence + replay across restart.
  • AgentSession.deliver — wake-on-idle schedules a turn; mid-turn buffers; turn-end hook chains a follow-up turn when the inbox is non-empty.
  • JSONL schema round-trips — every record type encode/decode; one explicit "v=2 adds an optional field" forward-compatibility test so future schema bumps don't break v1 reads.
  • MCP tools — aegis_enqueue and aegis_task_status: shape, error responses (unknown queue, missing payload), from_handle reflection.
  • aegis_handoff refactor — existing handoff tests continue to pass; new assertion that the delivered InboxMessage carries sender="agent:<from_handle>".
  • Config loader — valid queues dict parses; queue referencing unknown agent profile fails loud at load time with the offending queue name in the error.

Live integration

  • One new file tests/test_queue_live.py, auto-skips if claude is not on PATH. End-to-end: spawn a producer with a tiny opening prompt that calls aegis_enqueue for a trivial task, worker runs, callback arrives, producer reads the result. The smoke test that proves the whole loop is wired. Mirrors tests/test_integration_live.py.

Test data uses tmp_path for the state dir. No real Telegram, no network.

Non-goals

  • Replacing the vault job substrate. Cron/recurring schedules and migration of vault/+/jobs/ are vision Phase 6, not v1. The v1 queue is one-shot, agent-initiated work only.
  • Sync delegation tool. A wrapper that does enqueue + await in one MCP call is composable on top of v1 but not shipped.
  • Cancellation MCP tool. Cancel-via-inbox is sufficient for v1.
  • Dynamic queue creation. MCP plane does not mutate queue topology; queues are static config.
  • Worker pools / reuse. Workers are ephemeral; one task per spawn.
  • Priority / routing / load shedding. Strict FIFO per queue, no priority.
  • Cross-aegis-instance queues. V1 lives inside a single aegis serve process. Distribution is vision Phase 5.
  • TUI queue-status pane. Visibility in v1 is "queues are visible via the worker tabs they spawn". A dedicated queue/inbox observability pane is a follow-up polish item once the backend has proven out — it touches Telegram, MCP, and TUI so the data model has to settle first.
  • Long-lived bash terminals as senders. sender: terminal:<name> is reserved in the tag schema but not implemented; that primitive is vision Phase 4.
Open questions & deferred items
  • Worker exit signal. "Worker exits when its work is complete" assumes the worker calls aegis_meta to close, or hits a natural stop. The exact contract — does the substrate watch for an idle-with-no-pending-inbox state? a sentinel tool call? — is an implementation detail to settle in the plan. Likely the simplest answer: worker exits when its AgentSession closes (existing mechanism), and the substrate captures whatever the session's last assistant text was as result.
  • Result capture granularity. The worker's "final assistant text" is the most natural result, but workers that do many tool calls may have a long final message vs. a short one. V1 takes whatever the worker's last assistant turn produced verbatim; structured result envelopes are deferred.
  • Inbox file growth. Append-only JSONL grows unboundedly. V1 ships without compaction; a future rotation policy (rotate at N MB, keep last M files) is straightforward.
  • Two producers awaiting the same task_id. Not possible in v1 — only the original enqueuer receives the callback. If "subscribe to a task" becomes a need, it's an additive primitive.
  • Schema migration ergonomics. The v: 1 field plus accepting unknown-but-well-formed sender prefixes covers the easy cases; harder migrations (renaming a field, changing semantics) will need a one-off replay-and-rewrite script when they arrive.
