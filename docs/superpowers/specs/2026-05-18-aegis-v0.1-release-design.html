Goal

Ship the first proper tagged release of aegis: a polished README, a Material MkDocs site auto-deployed to GitHub Pages via CI, a green test CI badge, an MIT license, a v0.1.0 annotated tag, and a GitHub Release with notes. Everything shipped to date (stream-json CLI driver, full-screen Textual TUI, multi-tab + cross-tab signalling, generated handles, Ink theme engine, live token metrics, lazy start) becomes the 0.1.0 changelog.

Locked decisions

  1. Version v0.1.0 — matches the never-bumped pyproject; first tag (none exist).
  2. Git tag + GitHub Release only — no PyPI; install stays uv/from-source (it drives the local claude CLI).
  3. MkDocs Material, focused page set; the public site excludes docs/superpowers/ (specs/plans stay internal).
  4. MIT license.
  5. Docs deploy via mkdocs gh-deploy from a GitHub Actions workflow to the gh-pages branch (Material's recommended path; least config).

Versioning & changelog

  • pyproject.toml remains version = "0.1.0" (already correct — single source of truth).
  • aegis --version: a typer --version eager flag on the app, printing aegis <v> from importlib.metadata.version("aegis") then exiting. It must short-circuit before config loading / the TUI launch (works with no .aegis.py).
  • CHANGELOG.md (Keep-a-Changelog): a single ## [0.1.0] - 2026-05-18 section, grouped Added, listing the shipped capabilities (CLI driver, TUI, multi-tab, handles, themes/Ink, metrics, lazy start). Link-style compare footer omitted (no prior tag).

Shiny README (markdown)

README.md is rewritten (GitHub convention = markdown; the global HTML-spec rule does not apply to project README/docs). Sections, in order:

  • Title + tagline: “Aegis — a meta-harness for coding agents.”
  • Badge row: CI status (the ci.yml workflow), Python 3.13, License MIT, Docs (link to the Pages site). Shields.io static/dynamic badges; no external JS.
  • What & why: 3–4 sentences — drives Claude Code via claude -p stream-json, multi-agent TUI, the control plane above the harness.
  • TUI snapshot: a fenced ASCII block approximating the Ink TUI (tab bar, transcript, status line). No image generation (explicit-only policy; none requested).
  • Quickstart: uv pip install -e . → aegis init → aegis.
  • Keys / tabs / themes table (from the shipped keymap).
  • Config: the .aegis.py agents example.
  • Docs & vision links (Pages site; vision lives in the private vault — link the in-repo docs/architecture.md summary instead of the vault path).
  • Status & roadmap: phases shipped → next (Phase 3 MCP plane).
  • License: MIT.

MkDocs site

mkdocs.yml (repo root)

  • site_name: Aegis, site_description, repo_url: https://github.com/apiad/aegis, repo_name: apiad/aegis, edit_uri to docs/.
  • theme: name: material with palette (dark default, slate/amber to echo Ink), navigation features (navigation.sections, navigation.top, content.code.copy), and a code highlight.
  • docs_dir: docs with exclude_docs: |\n superpowers/ (mkdocs ≥ 1.5) so docs/superpowers/** (specs/plans) is never built or published.
  • nav: Home → Install → Usage → Configuration → Architecture & Vision → Roadmap.
  • Dependency: a docs dependency group in pyproject.toml (mkdocs-material) so the site builds reproducibly via uv locally and in CI.

CI/CD (.github/workflows/)

ci.yml — tests

On push and pull_request: checkout, install uv (astral-sh/setup-uv), uv sync, uv run pytest -q -k "not live" (the live claude test can't run on CI — no auth/binary; the marker already excludes it). Single job, Python 3.13. Its status feeds the README badge.

docs.yml — docs deploy

On push to main (and workflow_dispatch): checkout (full history for gh-deploy), install uv + the docs group, uv run mkdocs gh-deploy --force --no-history. Permissions: contents: write (push to gh-pages). Concurrency guard so overlapping pushes don't race the branch.

One-time Pages enablement (release step, not a workflow)

GitHub Pages source must be set to the gh-pages branch. Repo settings can't be toggled by hand here; use gh api -X POST repos/apiad/aegis/pages -f source[branch]=gh-pages -f source[path]=/ (or PUT if Pages already exists), run once after the first successful docs.yml deploy creates the branch. Captured as an explicit ordered step in the plan.

Release mechanics

  1. LICENSE file: MIT, Copyright (c) 2026 Alejandro Piad Morffis. In pyproject.toml use the table form license = {text = "MIT"} + the "License :: OSI Approved :: MIT License" classifier (table form, not the PEP 639 SPDX string, to avoid hatchling-version warnings).
  2. Annotated tag v0.1.0 on the post-merge main HEAD: git tag -a v0.1.0 -m "aegis v0.1.0", push the tag.
  3. gh release create v0.1.0 --title "aegis v0.1.0" --notes-file <(changelog 0.1.0 section) — notes are the CHANGELOG body for 0.1.0 plus a one-line link to the docs site.

Order & dependencies

  1. LICENSE + pyproject license/docs-group + --version flag + CHANGELOG.
  2. README rewrite.
  3. mkdocs.yml + the six docs/*.md pages; verify uv run mkdocs build --strict clean locally (catches broken nav/links and confirms superpowers/ is excluded).
  4. ci.yml + docs.yml.
  5. Commit & push main → docs.yml runs, creates gh-pages → enable Pages via gh api → confirm the site is live.
  6. Tag v0.1.0 + gh release create — last, so the release reflects the polished, docs-live state.

Steps 1–4 keep uv run pytest -q -k "not live" green at each commit (the --version flag is the only code change; tested).

Verification

  • --version: a typer CliRunner test — aegis --version prints aegis 0.1.0 and exits 0 with no .aegis.py present (must not require config or launch the TUI).
  • Docs build: uv run mkdocs build --strict succeeds locally and in a CI dry-run; assert the built site/ contains no superpowers path (exclusion holds). A tiny test or a documented manual check; the --strict build failing on warnings is the gate.
  • CI workflows: YAML lint / act not required; correctness verified by the first real run on push (the plan treats the first green Actions run + a reachable Pages URL as the acceptance check).
  • Full suite stays green (uv run pytest -q -k "not live") through steps 1–4; the live driver test is unchanged.
  • Test work written/validated inline (not delegated).

Non-goals

  • PyPI publishing / build artifacts.
  • API / module reference docs, contributing guide, versioned (mike) docs.
  • Release automation (auto-tag-on-merge); the tag/release is a deliberate manual step this round.
  • Screenshots / generated images (explicit-only policy).
  • Moving or publishing docs/superpowers/ (stays internal, excluded).
Deferred / future

PyPI + signed wheels, mike for versioned docs, an autorelease workflow on tag push, and an architecture diagram are natural follow-ups once the project stabilises past Phase 3.
