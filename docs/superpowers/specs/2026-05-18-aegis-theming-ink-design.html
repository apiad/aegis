Goal

Make the TUI prettier and themeable: ship a calm "Ink" default (near-black, single warm accent, hairline rules instead of boxes, generous padding, a blank-line gap between user and agent turns) on top of a theme abstraction so additional themes are later drop-ins with no rework. Reference mockups: .playground/aegis-mockups/{A,B,C}-*.html (C = Ink, chosen).

Locked decisions

  1. Textual-native themes. Each aegis theme is a textual.theme.Theme, registered on the app; TCSS uses theme variables; rich-renderable colors read semantic roles from the active theme.
  2. Ink is the default and only theme now. The registry/abstraction is built so a new theme is a drop-in Theme object; no config key, no picker, no runtime switch (those are non-goals).
  3. Colors threaded explicitly (pane → render / dot / widgets), not a module global — keeps render_event and AgentState pure-given-input and test-trivial.
  4. Turn spacing is content. The blank-line gap between turns is real blank rows written to the transcript; panel padding and hairline rules are TCSS.

Theme engine — new src/aegis/tui/themes.py

  • INK: a Theme(name="aegis-ink", dark=True, …) with the C palette — near-black background, slightly-lifted surface/panel, parchment foreground, one amber accent, muted success/warning/error; plus aegis keys in variables: aegis-muted, aegis-faint, aegis-rule.
  • THEMES: dict[str, Theme] = {"ink": INK} and DEFAULT_THEME = "aegis-ink" (the registered theme name). Adding a theme later = add one Theme + one registry entry; nothing else changes.
  • AegisColors — a frozen dataclass with exactly the semantic roles aegis's rich renderables need: ready, working, error, accent, muted, ok, err, user (each a hex string).
  • aegis_colors(theme: Theme) -> AegisColors — the single place role→color is decided: ready=success, working=warning, error=error, ok=success, err=error, accent=accent (user › prefix and tool ⏺ glyph), user=accent, muted=variables["aegis-muted"] (thinking, ── done ──, the └ result gutter). Reads theme.success etc. and theme.variables[...].

Wiring

  • AegisApp.__init__/on_mount: for each THEMES value self.register_theme(t); set self.theme = DEFAULT_THEME. Expose self.colors: AegisColors = aegis_colors(self.current_theme). Add def watch_theme(self, _) that recomputes self.colors and re-pushes it to live panes — the extensibility seam (correct now, exercised once a 2nd theme exists).
  • TCSS (app.py CSS, ConversationPane.DEFAULT_CSS, widgets.py TabBar.DEFAULT_CSS): replace ad-hoc values with theme vars — $background $surface $panel $foreground $accent $boost and the Ink look: no panel boxes; TabBar gets border-bottom: hkey $panel-style hairline (via the aegis-rule color), StatusBar a top hairline; RichLog gains padding: 1 2; Input a calm padding; no rounded boxes (Ink = ruled, not boxed).
  • state.py: replace the AgentState.dot property with dot(self, colors: AegisColors) -> str returning f"[{colors.ready}]●[/]" etc.; keep label unchanged.
  • render.py: render_event(ev, colors: AegisColors) — replace the literal "cyan"/"green"/"red"/"dim" styles with colors.accent / colors.ok / colors.err / colors.muted. SystemInit/Unknown → None unchanged.
  • ConversationPane: receives the active AegisColors from the app (constructor arg or set right after mount); passes it to render_event and to StatusBar/_TabCell dot rendering. The user › prefix uses colors.user.
  • widgets.py: StatusBar and _TabCell take colors for the dot + accent slug; TabBar.set_tabs tuple grows no fields — the app passes colors to TabBar once (it owns the cells).
  • cli.py: unchanged (the app already owns theme setup).

Turn spacing

In ConversationPane: before writing a new user › line (except the very first turn), write one blank line to the transcript; after the assistant block / ── done ── separator, the next user turn's leading blank line provides the gap (so exactly one blank row between turns, none stray at top). Panel padding (the "padding everywhere") is TCSS on RichLog/Input; the calm Ink whitespace is the combination of both.

Edge cases

  • A registered Textual theme name must be unique; INK.name is "aegis-ink" (namespaced) to avoid clashing with Textual's built-ins.
  • aegis_colors reads theme.variables by key; a missing aegis key falls back to theme.foreground (so a future under-specified theme degrades, not crashes).
  • watch_theme may fire before panes exist (initial set in on_mount) — it must no-op safely when there are no panes.
  • Colors are hex strings consumed by Rich markup ([#RRGGBB]…[/]) and TCSS; both accept #RRGGBB.

Testing

  • themes.py (unit): THEMES contains the default; INK.name == "aegis-ink"; aegis_colors(INK) returns an AegisColors with every role a non-empty #-hex; the fallback path (synthetic theme missing an aegis variable) yields foreground, no raise.
  • state.dot(colors): each state embeds its Ink hex (success/warning/error); label unchanged.
  • render_event(ev, colors): tool glyph uses accent hex, ok/err use ok/err, thinking/result use muted; markdown text still rendered; SystemInit/Unknown → None.
  • Pilot: app boots with app.theme == "aegis-ink" and app.palette is the Ink mapping (the app attribute is palette, not colors, to avoid shadowing Textual's App.colors); after a user turn the transcript contains a blank line separating the user line from the agent reply; tab/status render with Ink colors.
  • Existing pilots/units updated for the new dot(colors) / render_event(ev, colors) signatures (mechanical). The live claude driver test is unchanged (driver-level).
  • Test work written/validated inline (not delegated).

Non-goals

  • Config (.aegis.py) theme selection; a theme picker; a runtime theme-switch key.
  • The A/B (Warm Parchment / Cool Slate) palettes — later drop-in Theme objects using this same engine.
  • Light themes, per-tab themes, user-defined themes from config.
Why threaded colors, not a module global

A module-level "current palette" set by the app would avoid signature churn but reintroduces global mutable state that tests must set up and reset and that prior reviews in this project flagged. Threading AegisColors keeps render_event and AgentState.dot pure functions of their inputs — the same property that made them trivially unit-testable in earlier phases.
